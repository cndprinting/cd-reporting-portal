# Original setup guide follows

This is the step-by-step to get the C&D Reporting Portal receiving real-time USPS
Intelligent Mail barcode (IMb) scan data — the foundation for Accutrace-style tracking.

> **Timeline:** Application + provisioning takes **1–3 weeks** from USPS. Start this now.

---

## What we're applying for

**Informed Visibility® Mail Tracking & Reporting (IV-MTR)** — USPS's official
real-time feed of scan events for mailpieces carrying an IMb or IMpb.

- Free from USPS (no per-piece fee for the data itself)
- Delivered via SFTP push, S3 push, or REST/pull API
- Requires BCG account + Mailer ID (MID) + CRID

---

## Step 1 — Prerequisites (should already exist)

### a. Business Customer Gateway (BCG) account
- URL: https://gateway.usps.com
- Owner: must be a C&D person with legal authority (e.g., Benjy or the GM)
- If C&D already mails commercially, the account exists — find the credentials
  from whoever handles mail drops (likely the mailshop operations lead)

### b. Customer Registration ID (CRID)
- Assigned by USPS to a business location. C&D has at least one tied to the
  Daytona facility.
- Find it: BCG → Manage Account → Business Services → "My Profile"

### c. Mailer ID (MID)
- 6-digit (high-volume, ≥1M pieces/yr) or 9-digit (default)
- Find in BCG → Mailer ID service
- If C&D mails through a partner/mail shop that uses *their* MID, we need to
  either (a) get our own, or (b) request IV-MTR delegation from the partner

---

## Step 2 — Request IV-MTR access

1. Sign into BCG: https://gateway.usps.com
2. Click **Manage Services** → **Request Service**
3. Select **Informed Visibility — Mail Tracking & Reporting**
4. Fill the service request:
   - **Business Location (CRID):** the C&D Daytona CRID
   - **Users who need access:** list at minimum Benjy + one engineer/admin
   - **Data type:** Mail tracking (scans on IMb letters/flats)
   - **Data volume estimate:** "X million pieces/year" (use C&D's actual annual
     volume; USPS uses this to size your feed)
5. Submit and wait for email approval from USPS IV Help Desk

**Typical approval: 5–10 business days.** If it stalls, call USPS Mailing &
Shipping Solutions Center: **1-877-672-0007**.

---

## Step 3 — Configure the data feed

Once approved, IV-MTR admin in BCG unlocks. In there:

### Create a "Data Delegation" (so the portal can receive scans)

1. IV-MTR → **Data Delegation** → **Create New**
2. **Data Recipient:** same CRID (we're recipient of our own data)
3. **Data Source:** your MID(s) — include every MID C&D uses
4. **Mail Type:** "All Mail" (Letters + Flats for our use case)
5. **Start Date:** today
6. **Frequency:** Near-real-time (every 15 min) — this is the Accutrace-equivalent

### Pick a delivery method (pick ONE)

**Option A — S3 Push (recommended, lowest latency)**
- USPS drops JSON files into an S3 bucket you control, every 15 min
- You give USPS an IAM role ARN they can `s3:PutObject` to
- Set up a Lambda (or Vercel serverless function webhook) that fires on S3 PUT
  and POSTs the file to our `/api/iv-mtr/ingest` endpoint with the
  `x-iv-mtr-key` header

**Option B — SFTP Push**
- USPS drops CSV files into an SFTP server you control
- Run a small cron/worker that pulls new files and POSTs to `/api/iv-mtr/ingest`
- Works fine but slightly more moving parts

**Option C — Pull via REST (what our `/api/iv-mtr/pull` cron already does)**
- Our Vercel Cron hits `/api/iv-mtr/pull` every 30 min
- That route logs into IV-MTR, requests the last 60 min of scans, ingests them
- Good to get running day 1; migrate to S3 push once volume grows

## Step 4 — Environment variables

Set these in Vercel → Project → Settings → Environment Variables
(Production + Preview):

```bash
# Shared secret for the ingest endpoint (rotate periodically)
IV_MTR_INGEST_KEY="<generate a 32-char random string>"

# For the pull-mode cron
IV_MTR_API_BASE="https://iv.usps.com/ivws/api"
IV_MTR_USER_ID="<BCG username>"
IV_MTR_PASSWORD="<BCG password>"
IV_MTR_MID="<6 or 9 digit MID>"

# Vercel cron authorization (auto-generated by Vercel, or set your own)
CRON_SECRET="<random string>"
```

> **Security:** Use Vercel's encrypted env vars only — never commit these.
> IV_MTR credentials are tied to a real USPS login; treat them like banking creds.

## Step 5 — Prep the mail file pipeline

For scans to be useful, every mailpiece the portal tracks needs a row in
`MailPiece` table *with its IMb* before the drop hits USPS. Workflow:

1. **Campaign setup (portal UI)** — create Campaign + MailBatch
2. **Generate IMbs** — our estimator/mailshop assigns a unique serial per piece
   - For each piece, combine: BC (00 or 01) + STID (300/700/etc.) + MID + serial + routing ZIP
   - Helper in `src/lib/services/imb.ts` → `generateSerials()` + `buildIMb()`
3. **Import mail file** — POST to `/api/mail-pieces/import` with campaignId,
   mailBatchId, and the CSV rows (recipient + IMb)
4. **Drop the mail** — our mailshop prints the barcodes and delivers to USPS
5. **Scans start flowing 6–24h later** — first scans appear when USPS accepts
   the mail at the induction facility

---

## Step 6 — Verify (seed list)

Include **20–50 "seed" addresses** in every mailing — addresses at known
locations (employees, friends, office) marked with `isSeed: true`. When the
scans arrive:

- Seed pieces should hit ORIGIN_ACCEPTANCE within 24h
- Seeds should show OUT_FOR_DELIVERY and DELIVERED within the expected window
- If a seed doesn't scan, something in the ingestion pipeline is broken
  (bad IMb format, wrong MID, S3 permissions, etc.)

The Mail Tracking page already highlights seed pieces with a badge.

---

## Step 7 — Monitoring

- **`IVFeedIngestion` table** logs every ingest batch with counts + errors.
  Admin page should show last N ingestions at a glance.
- **Alert if no ingestion in >1 hour** — USPS outages or credential issues.
  Set up a Vercel cron + email via Resend.
- **Reconcile monthly** — compare `MailPiece.status = DELIVERED` count against
  USPS PostalOne! reports. Should match within ~2% (accounts for missed final
  scans on ~3% of mail).
